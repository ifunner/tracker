GO Schedule PWA — Design Kit

A mobile-first Progressive Web App for searching GO Transit (Toronto) train & bus schedules, optimized for the daily commuter who reuses one or two trips. This document is the single source of truth for engineering: design tokens, component specs, interaction rules, data shape, and the decisions still pending sign-off.

Platform · Mobile web / installable PWA (~390px) Scope · v1 direct trips, single line Status · Visual locked · data layer pending
How to read this kit

Values marked LOCKED are extracted directly from the approved interactive component (GO Schedule.dc.html) and are build-ready. Items marked DECIDE still need your sign-off — each carries a recommended default so dev is never blocked. Section 11 collects every open decision in one checklist.

01 — Product Principles

  • Returning user first. Opening the app shows next departures for saved trips instantly — zero taps, no spinner blocking the answer.
  • Departure time is the hero. Times use the condensed display face at large sizes; everything else is supporting metadata.
  • Line color accents, never floods. Brand color appears as a badge or a thin rail — surfaces stay neutral.
  • Tap-friendly & offline-resilient. ≥44px primary targets; the schedule index must work fully offline.

Dark-mode brightening rule

For non-badge uses (rails, dots, hero countdown numerals), blend the brand hex 34% toward white in dark mode only. Badges never brighten.

lineAccent = theme === 'dark'
  ? mix(brandHex, '#ffffff', 0.34)   // rails, dots, numerals
  : brandHex;
// e.g. BR #003767 → ~rgb(86,121,151) on dark

06 — Component Specs LOCKED

Line badge

Filled brand-color chip, white Barlow SC 700 text 11–13px, padding 3×9px, radius 6–7px, letter-spacing .6px. Two-letter code. Never brightened; white text on all seven lines passes contrast.

Saved trip card

Two approved looks. DECIDE which ships as default (§11) — recommend Board for at-a-glance scanning.

  • Board: header (badge · O→D · reverse) → hero time 46px + accent "in X min" pill + mm:ss/arr/duration line → divider → 2 secondary departure rows → "Show more" reveals 2 more.
  • Rail: 6px line-accent rail at left edge → badge + line name → O→D → hero countdown 36px in line-accent → next departures as tappable time chips → "Show more".
  • Tapping the card body toggles expand. Tapping any departure (hero or row/chip) opens Trip Detail. The reverse button stops propagation.

Departure row · result row

Result row: left line-accent bar (4px) · dep time 27px → arr 21px · badge + headsign + stop count · right column with "in X min" (accent) over duration over a save toggle (★/☆, 32px). The two soonest rows get an accent-tinted border. Whole row is tappable to detail; save button stops propagation.

Inputs

  • O/D field: grouped card; From has a hollow ring marker, To a filled square; a swap button overlaps the divider on the right. Tapping a field opens the full-height station picker.
  • Station picker: slide-up panel, search input + filtered list; each row = line-color dot + name + code. Union shows a neutral #8b949a dot + "HUB".
  • Date / Time: two native inputs (themed via color-scheme), plus a "Leave now" pill that resets both to the current moment.

Tab bar · toast · cues

  • Tab bar: Home / Search / Saved; active = accent fill + accent label, inactive = --text-3. Results & Detail are pushed views that keep Search (or originating tab) lit.
  • Toast: pill, --text bg on --bg text, bottom-center, auto-dismiss 2.2s.
  • Offline / install cues: "Offline ready" pill in the Home header; a dismissible "Install GO Schedule" banner. Decorative in the mockup — see §10.

08 — Interaction & Behavior Rules

Countdown logic LOCKED

mins = ceil((depart - now) / 60s)
inLabel:  mins <= 0          → "now"
          mins  < 60         → "in {mins} min"
          else               → "in {h}h {mm}m"
mmss:     secs < 3600        → "M:SS"  (ticks every 1s, soonest only)
          secs >= 3600       → falls back to inLabel
clock tick: setInterval 1000ms re-renders all live values

DECIDE: confirm the mm:ss seconds ticker (vs minute-only), and how many decimals of "live" you want once real-time (GTFS-RT) deltas exist.

Departures shown LOCKED

Collapsed card = 1 hero + 2 rows. Expanded = +2 (5 total). Results list fetches the next 9 from the reference time. DECIDE: 2 vs 3 secondary rows as the collapsed default.

Core actions LOCKED

  • Reverse: swaps O↔D in place and persists (morning ↔ evening commute on one card). DECIDE: persist vs momentary peek.
  • Save: from a result row; de-duplicates, appends to saved, toasts "Saved to Home". Star toggles ☆→★.
  • Theme: manual toggle in the status bar; instant, no reload.
  • Search default: "Leave now" (depart-after). DECIDE: add an "arrive by" mode?

Navigation model LOCKED

Three tabs (Home / Search / Saved) + two pushed views (Results, Detail) with a back affordance. Back: Detail → originating screen; Results → Search. Search context (O/D, time) persists across the flow.

10 — Data Model & GTFS Mapping

The mockup runs on hardcoded sample data shaped as below. The production data layer turns the 104 MB GTFS feed into a compact per-day-type / per-station-pair index in IndexedDB — the UI contracts shouldn't change.

Line     { code, name, color }                 // 7 lines, §03
Station  { name, lineCode, stopId, code }       // name + line-color dot
StopSeq  [ [stopName, offsetMinutes], … ]       // ordered toward Union
Departure{ depart:epochMs, arrive:epochMs, travelMin }
SavedTrip{ origin, dest }                        // reverse = swap
Amenity  string[]                                // per origin station

// derived
travelMin   = |destOffset - originOffset|
arrive      = depart + travelMin*60s
stopsBetween= |destIdx - originIdx| - 1
headsign    = line terminus in travel direction
  • DECIDE — service calendars: weekday / Saturday / Sunday / holiday day-types must drive which departures exist. Mockup uses a flat 30-min headway.
  • DECIDE — static vs real-time: scheduled GTFS only, or GTFS-RT for delays / "on time" badges? Large scope delta.
  • DECIDE — transfers: v1 is direct-only (cross-line searches currently clamp to Union). "Connection at Union" is a separate effort.
  • DECIDE — amenities: source the real shelter/washroom/bike/bench flags from the feed; pick an icon set (mockup uses text chips).

11 — PWA & Accessibility

  • Manifest + icons: not in the mockup. Need name, theme/background color (tie to --bg), maskable icon set, standalone display.
  • Service worker / caching: app shell + schedule index cached for offline; real-time data (if any) network-first with stale fallback.
  • Theme: manual toggle done; DECIDE auto/device-follow (prefers-color-scheme).
  • Contrast: verify all line-accent-on-surface text meets WCAG AA in both themes; the brightening rule (§03) exists for this — re-check after final hex sign-off.
  • Targets & focus: 44px minimum; the symbol glyphs (⇅, ★, ←, ×) need ARIA labels and visible focus rings for keyboard/AT.
  • Reduced motion: respect prefers-reduced-motion for the slide/fade transitions and the ticking countdown.
